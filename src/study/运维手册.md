## docker迁移

> [!NOTE]
>
> docker迁移文件：镜像、volume、docker-compose.yml

如果你想将一个或多个 **Docker 容器打包成 `.tar` 文件**，实际的操作是：

- 使用 `docker commit` 把**容器保存为镜像**

```
docker commit <容器名或ID> <新镜像名>:<标签>
```

- 使用 `docker save` 把镜像导出为 `.tar` 文件

### 迁移步骤：

1. 查看正在运行的容器对应的镜像 

   ```shell
    docker ps --format "table {{.Names}}\t{{.Image}}"
   ```

2. 导出为 `.tar` 文件  

   ```bash
   docker save -o test.tar test
   ```

3. 把volume、docker-compose.yml同步

   本地执行同步到服务器

```scala
rsync  -av   /test/demo/     root@192.168.120.129:/test/demo
```

> [!TIP]
>
>  本地执行从服务器同步到本地

rsync  -av   root@192.168.1.1:/home/ar/data/tmp/  /test/tmp/

## Docker本地文件加载

Docker `load -i` 使用详解及结合 Docker Compose 启动案例

🧠 一、`docker load -i` 详解

✅ 命令格式：

```bash
docker load -i <image_file.tar>
```

📌 参数说明：

- `load`: 加载一个镜像存档（通常通过 `docker save` 生成）。
- `-i`: 指定 `.tar` 镜像文件的路径。

---

📦 二、结合 `docker-compose` 启动项目

📝 小提示

- 使用 `docker images` 查看当前系统镜像。
- 确保 `docker-compose.yml` 中的 `image` 名称与加载后的镜像名称一致。

✅ 示例目录结构

```bash
project/
├── docker-compose.yml
├── my_image.tar
```

✅ 步骤

1. 加载镜像

```bash
docker load -i my_image.tar
```

2. 编写 `docker-compose.yml`

```yaml
version: '3'
services:
  app:
    image: my_image:latest
    ports:
      - "8080:80"
    restart: always
```

3. 启动项目

```bash
docker-compose up -d
```

## Docker迁移方案

用 `rsync`，先全量 → 再增量同步 → 切换前停容器 + 最后同步。

第一步：全量迁移。本地执行，从服务器同步到本地

`-a` 保留权限、符号链接等

`-v` 显示详细过程

`-z` 压缩传输

`--progress` 显示进度

```
rsync -avz --progress root@192.168.1.1:/home/ar/data/tmp/  /home/ar/data/tmp/  
```

第二步：切换前再做一次增量迁移

两天后，切换前再执行一次相同命令：

```
rsync -avz --progress root@192.168.1.1:/home/ar/data/tmp/  /home/ar/data/tmp/  
```

这次只会同步变化过的文件，非常快。

------

第三步：停容器 + 最终同步

为了保证数据一致性，在正式切换的那一刻：

1. 先停掉旧服务器上的相关容器（避免数据写入）：

   ```
   docker compose down
   # 或者 docker stop 容器名
   ```

2. 再执行最后一次同步（确保完全一致）：

   ```
   rsync -avz --progress root@192.168.1.1:/home/ar/data/tmp/  /home/ar/data/tmp/  
   ```

3. 在新服务器上启动容器。

这样就能保证数据无丢失地迁移过去。

## 迁移问题处理

 **MongoDB 的 WiredTiger 存储引擎启动失败日志**，核心问题是 **数据文件或日志损坏**，导致 MongoDB 无法启动

### MongoDB无法启动

#### 1️⃣ 停止容器

先强制停止正在重启的容器：

```
docker stop b649be210d2f
```

- 停止后容器不会自动重启。

------

#### 2️⃣ 备份数据

在宿主机上备份数据目录 `/home/ar/data/demo/mongo/data`：

```
cp -r /home/ar/data/demo/mongo/data /home/ar/data/demo/mongo/data_backup
```

- 这样即使修复失败，也不会丢失数据。

------

#### 3️⃣ 临时创建一个修复容器

使用 `--repair` 修复数据库，不直接用原容器，因为原容器启动会不断重启。

```
docker run --rm \
  -v /home/ar/data/demo/mongo/data:/data/db \
  mongo:4.4.17 \
  mongod --dbpath /data/db --repair
```

- `--rm` 表示修复完成后容器会自动删除。
- 注意替换 `mongo:最新版本` 为你使用的镜像版本。
- 

#### 4️⃣ 启动原容器

修复完成后，重新启动原容器：

```
docker start b649be210d2f
docker logs -f b649be210d2f
```

- 查看日志确认 MongoDB 能正常启动。

## 开放防火墙端口

```

重新加载防火墙配置，使刚刚通过 `--permanent` 添加的规则立即生效。

```

🔍 逐项详解：

| 参数                  | 含义                                                         |
| --------------------- | ------------------------------------------------------------ |
| `firewall-cmd`        | firewalld 提供的命令行工具，用于管理防火墙规则               |
| `--zone=public`       | 指定要修改的防火墙区域（zone），此处为 `public`（公共区域）  |
| `--add-port=3306/tcp` | 表示要打开端口号 3306，使用 TCP 协议                         |
| `--permanent`         | 将更改永久保存（写入配置文件），否则只会临时生效，重启后失效 |

#### docker 中运行的mysql、redis、mongodb、minio 有数据集映射。怎么改密码

```sh
mysql  
不要用 MYSQL_ROOT_PASSWORD 重新起容器，这对已有数据无效
ALTER USER 'root'@'%' IDENTIFIED BY '新密码';
FLUSH PRIVILEGES;

    command: redis-server --requirepass ar.redis147

MongoDB
use admin
db.changeUserPassword("root", "新密码")

MinIO

  -e MINIO_ROOT_PASSWORD=新密码 
```

| 服务    | 是否能在线改 | 是否影响 volume | 正确方式               |
| ------- | ------------ | --------------- | ---------------------- |
| MySQL   | ✅            | ❌               | `ALTER USER`           |
| Redis   | ✅            | ❌               | `CONFIG SET` / command |
| MongoDB | ✅            | ❌               | `changeUserPassword`   |
| MinIO   | ❌            | ❌               | 重启容器改 env         |

##### mysql改了密码后mysqldump备份还是旧密码

只改了 `root@'%'` 的密码，但 **`mysqldump` 实际用的是 `root@'localhost'`**，而这个账号 **密码还是旧的**，所以只有旧密码能用。

```sh
ALTER USER 'root'@'localhost' IDENTIFIED BY '新密码';
FLUSH PRIVILEGES;
```

## Mysql备份

mysqldump+binlog　备份docker 中mysql8.0教程

宿主机执行

```bash
docker exec mysql80 bash -c 'mysqldump --single-transaction --flush-logs --source-data=2 --routines --opt -uroot -proot123 -h192.168.120.128 -P3306 mysql_test > /var/lib/mysql/full_backup.sql'
```

> [!NOTE]
>
> 如果数据卷没做映射将sql拷贝到宿主机   docker cp mysql80:/full_backup.sql ./full_backup.sql

### 参数说明：

- `--single-transaction`：一致性备份。
- `--flush-logs`：刷新 binlog，确保 binlog 编号唯一。
- `--master-data=2`：写入当前 binlog 文件名和位置。
- `-h192.168.120.128 -P3306` 宿主机ip和端口

查看 `full_backup.sql` 开头内容：

```
-- CHANGE MASTER TO MASTER_LOG_FILE='mysql-bin.000003', MASTER_LOG_POS=1234;
```

这就是 binlog 的起点，用于 **基于时间点恢复（PITR）**。

## 恢复逻辑备份 + binlog

（1）恢复 `mysqldump`：

```
docker exec -i mysql80 mysql -uroot -proot123 -h192.168.120.128 -P3306 mysql_test < /test/tmp/volume/data/full_backup.sql
```

（2）使用 `mysqlbinlog` 工具

恢复 binlog 到某个时间点 在docker 容器中解析

```
mysqlbinlog --no-defaults --base64-output=DECODE-ROWS --start-datetime="2025-07-03 10:00:00" \
            --stop-datetime="2025-07-03 12:40:00" \--verbose /var/lib/mysql/binlog.000002 > /var/lib/mysql/decoded.sql
```


### 前提

1. 你的 Elasticsearch 版本是 **7.17.4**（插件版本必须匹配）。
2. 你已经下载了 `elasticsearch-analysis-ik-7.17.4.zip` 到本地。
3. Docker 容器已经有 `volumes` 映射：

```
volumes:
  - ./plugins:/usr/share/elasticsearch/plugins
```

### 解压插件到本地

在宿主机上执行：

```
mkdir -p ./ik
unzip elasticsearch-analysis-ik-7.17.4.zip -d ./ik
```

**注意：** 解压后的目录结构应该类似：

```
./plugins/ik/
├── ik/
│   ├── plugin-descriptor.properties
│   ├── ...
```

Elasticsearch 插件目录里，**每个插件必须在独立目录下**，比如 `ik`，不能直接解压到 `plugins/` 根目录。

------

### 重启 Elasticsearch 容器

插件安装完成后，**必须重启 Elasticsearch** 才能生效：

```
docker restart elasticsearch
```

### 验证插件安装成功

进入容器：

```
docker exec -it <elasticsearch-container-name> /bin/bash
然后运行：
/usr/share/elasticsearch/bin/elasticsearch-plugin list
输出应该包含：
ik
```

### `interfaces` 文件（适用于传统 Debian 配置）

#### 配置网卡

```
sudo nano /etc/network/interfaces
```

添加或修改为类似以下内容（以 `ens33` 为例）：

```
auto ens33
iface ens33 inet static
    address 192.168.120.130
    netmask 255.255.255.0
    gateway 192.168.120.2
    dns-nameservers 8.8.8.8 223.5.5.5
```

保存后重启网络服务：

```
sudo systemctl restart networking
```

#### 配置DNS

```
vi /etc/resolv.conf
```

> [!NOTE]
>
> 网关与配置的DNS服务器需一致

## [时间同步](https://blog.csdn.net/PEABRAIND/article/details/124385949)

✅ **方法一：临时禁用自动同步再设置时间**

如果你确实需要手动设置时间，可以先禁用自动同步：

```bash
timedatectl set-ntp false   # 关闭自动时间同步
timedatectl set-time '2025-08-07 14:00:00'  # 设置你需要的时间
debain sudo date -s '2025-08-29 18:49:00' && sudo hwclock -w     # 永久 修改系统时间 + 写入硬件时钟
timedatectl set-ntp true    # 设置完时间后重新开启自动同步（可选）
```

------

✅ **方法二：修改 NTP 设置（如果你想保持自动同步）**

如果你只是想让系统时间正确，可以通过 NTP 服务自动同步，而不是手动设定：

```bash
# 查看当前时间同步状态
timedatectl status

# 手动同步一次（如果使用 chrony）
chronyc makestep
```
